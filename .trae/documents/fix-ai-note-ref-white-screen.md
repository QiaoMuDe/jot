# 修复 AI 引用笔记选择器白屏问题的终极方案

## 摘要

用户反映引用笔记选择器仍有白屏，且查询速度慢。深度对比搜索弹窗和 AI 引用浮层后，发现核心差距不在查询路径，而在于**动画延迟、CSS 约束和防抖策略**三重叠加。

## 根源分析

### 元凶一：入场动画延迟（白色屏幕的根源）

对比两个组件的 item 动画：

| 组件 | 动画 | 延迟 | 20 条总耗时 |
|------|------|------|------------|
| 搜索弹窗 | 无（`search-modal-result-item` 无动画） | 0ms | **即时渲染** |
| AI 引用浮层 | `ref-item-in 0.25s ease-out both` + `--i * 30ms` | 每项 30ms | **20×30ms=600ms** 才完全可见 |

流程：
1. `appendToList()` 将 20 条项目插入 DOM → **项目有高度但 opacity=0（不可见）**
2. 加载器隐藏 → 用户看到空白区域
3. 每 30ms 一项淡入 → 600ms 后才全部可见

**这 600ms 的不可见期正是用户看到的"白屏"**。

### 元凶二：max-height: 360px 硬编码限制

`.ai-note-ref-list-wrap` 的 `max-height: 360px` 意味着列表区域最多显示 ≈6 条项目（60px/条）。每页加载 20 条，用户需要滚动 3 屏才能看完一页的内容。这导致：
- 滚动触发时机异常（400px 阈值在 360px 容器内无法正常工作）
- 追加后需要跳转"不可见区域"才能看到新内容

### 元凶三：防抖 300ms 比搜索弹窗 200ms 多 50%

搜索弹窗搜索输入防抖 200ms，AI 引用浮层搜索输入防抖 300ms。感知延迟差异明显。

### 元凶四：加载器设计缺陷

当前加载器是 `refListWrap` 的子元素（和 `refList` 同级），位于列表下方。当新条目到达时：
1. 条目先插入 `refList`（有高度但不可见，因动画延迟）
2. 加载器隐藏
3. 条目在 600ms 内逐渐淡入

用户看到：加载器消失 → 空白区域 → 条目慢慢出现。

## 改动方案

### 1️⃣ 追加模式禁用入场动画

**修改文件**：`frontend/src/js/ai-chat.js` — `appendToList()`

**做法**：追加的条目加上 `style="animation: none; opacity: 1; transform: translateY(0);"` 或添加无动画的 CSS 类 `.ai-note-ref-item-append`。

更简洁的方式：在 `appendToList` 生成的 HTML 中直接内联覆盖：

```js
// 追加的 item 不加 animation-delay 延迟
return `<div class="ai-note-ref-item${isSelected ? ' selected' : ''}" data-id="${note.id}" style="--i:${startIdx + idx};animation:none;opacity:1;transform:translateY(0)">
```

### 2️⃣ 移除 max-height: 360px 限制

**修改文件**：`frontend/src/css/components/ai-chat.css`

**做法**：`.ai-note-ref-list-wrap` 的 `max-height: 360px` 改为 `max-height: 50vh`（或完全删除，让 flex: 1 自然约束）。

这样列表可以撑满父容器，显示更多条目，滚动触发更正常。

### 3️⃣ 搜索防抖降至 200ms

**修改文件**：`frontend/src/js/ai-chat.js`

**做法**：引用笔记选择器的搜索输入防抖从 300ms 改为 200ms，与搜索弹窗一致。

### 4️⃣ 加载器改为固定高度占位

**修改文件**：`frontend/src/css/components/ai-chat.css`

**做法**：`.ai-note-ref-list-loader.visible` 时设置 `min-height: 80px` 或 `padding: 32px 16px`，让加载器占据更多垂直空间，减少视觉空白。

同时，在 `appendToList` 隐藏加载器前先插入条目，保证条目数据到达后立即占据位置，加载器是最后隐藏的——与当前实现一致，但配合移除动画后，条目会立即可见。

### 5️⃣（可选）搜索弹窗也改滚动阈值

**修改文件**：`frontend/src/main.js`

搜索弹窗滚动阈值也改为 400px 保持一致性。

### 改动影响评估

| 改动 | 影响范围 | 风险 |
|------|----------|------|
| 追加禁用动画 | 追加模式视觉（无动画但即时可见） | 低——初始渲染动画不受影响 |
| 移除 max-height:360px | 浮层最大高度取决于父容器 | 低——父容器有天生的 vh 限制 |
| 防抖 300→200ms | 搜索响应更快 | 低——与搜索弹窗一致 |
| 加载器高度增加 | 加载时多占一些空间 | 极低 |

## 验证步骤

1. **白屏测试**：打开引用笔记选择器 → 快速滚动到底部
   - 加载器应清晰可见
   - 数据到达后条目**立即显现**（无 600ms 淡入等待）
   - 无白屏闪烁

2. **速度测试**：切换笔记本筛选或输入关键词搜索
   - 防抖 200ms → 搜索响应更快
   - 列表容器高度更大 → 可显示更多条目

3. **视觉一致性**：
   - 初始打开时骨架屏正常
   - 首次渲染的条目有动画（保留 `ref-item-in`）
   - 只有追加的条目无动画
